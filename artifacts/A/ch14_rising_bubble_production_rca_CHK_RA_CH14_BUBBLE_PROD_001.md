# CHK-RA-CH14-BUBBLE-PROD-001: production-path root cause audit

## Scope

This audit revisits the rising-bubble blow-up using saved production artifacts,
after the common mass-momentum flux theorem was verified offline.  The purpose
is to identify which production variables actually enter the failure loop.

The analysis rejects damping, CFL-only reduction, pressure masking, curvature
smoothing, density clipping, and benchmark-specific branches as remedies.  A
valid explanation must be stated as a discrete conservation/energy failure.

## Theoretical hypotheses

H1. The blow-up is caused by insufficient CFL.

Rejected as root cause.  The time step stays near the capillary limit
`7.22e-06` until the energy has already started growing.  The advective limiter
collapses only after velocities become unphysical.  CFL is a symptom sensor,
not the generating mechanism.

H2. Volume loss is the primary cause.

Rejected as root cause.  At the end of the failed run the reported volume drift
is about `4.44e-03`, but the blow-up begins while divergence and volume-related
diagnostics are still small.  Reinitialization volume delta remains around
`1e-10` per step.  Volume handling can amplify geometry error, but it does not
explain the first kinetic-energy injection.

H3. Reinitialization is the primary cause.

Partially rejected.  The pure-transport exact-map and common-flux gates show
the same failure class without reinitialization.  In production, reinitialization
is a secondary contributor because it changes `psi` and hence `rho` without a
corresponding conservative momentum retraction.

H4. Raw curvature or capillary jump is the primary cause.

Rejected as first cause.  Near onset, `capillary_jump_linf` remains only
`O(10^3)`, while the capillary face acceleration grows from `O(10^3)` to
`O(10^12)`.  A moderate jump being mapped to a huge face acceleration indicates
an operator/history amplification problem, not merely a large Young-Laplace
jump.

H5. PPE solver nonconvergence is the primary cause.

Rejected as first cause.  The saved diagnostics show the defect-correction PPE
stays at its configured iteration cap with small final relative residuals; the
PPE RHS itself becomes enormous before the solver can be blamed.  The source
fed to PPE is already nonphysical.

H6. Face-native affine pressure history is a harmless IPC bookkeeping device.

Rejected.  The production path stores full pressure-acceleration face cochains
as history and injects them into the next step:

- `src/twophase/simulation/ns_pipeline.py:735` restores
  `previous_pressure_accel_face_components`.
- `src/twophase/simulation/ns_step_services.py:504` reconstructs previous face
  pressure acceleration to nodes and subtracts it in the predictor.
- `src/twophase/simulation/ns_step_services.py:953` adds the divergence of the
  previous face acceleration to the PPE RHS.
- `src/twophase/simulation/ns_step_services.py:1025` computes the next full
  pressure face cochain.
- `src/twophase/simulation/ns_pipeline.py:1003` stores it as the next history.

This is an acceleration-history loop.  It is not a conservative transport of
`rho u` by the same mass flux that moved `rho`.

H7. IMEX/BDF2 velocity history is harmless because BDF2 is stable.

Rejected in variable-density form.  BDF2 can be stable when written in a fixed
or properly controlled mass metric.  The current production path uses velocity
history in the current `rho` metric; the saved pre-blowup checkpoint shows this
history has become energetically explosive.

H8. The common mass-momentum flux remedy remains the right target.

Supported.  The offline common-flux theorem proves that a conservative update of
`rho` and `m=rho u` by the same mass flux is non-energy-increasing for pure
transport.  The production path fails exactly by using acceleration/velocity
history rather than a conservative common momentum flux.

## Identified production mechanism

The production failure is a coupled history-feedback form of the previously
identified mass-momentum transport inconsistency:

```text
psi transport/reinit  -> rho^{n+1}
velocity BDF2/EXT2    -> old-map velocity history in M(rho^{n+1})
affine face IPC       -> previous pressure acceleration face cochain
PPE RHS               -> div(previous face acceleration) + predictor RHS
new pressure faces    -> stored as next previous pressure acceleration
```

This loop is not written in conservative momentum variables and is not driven by
the same mass flux that updates `rho`.  Therefore the discrete kinetic-energy
identity has no theorem protecting it.  Once a small phase/momentum mismatch
appears near the moving high-density interface, the acceleration-history loop
can amplify a moderate capillary jump into huge face accelerations and then into
unphysical velocities.

The immediate production signature is the blow-up of
`previous_pressure_accel_face_components` and `capillary_face_linf`; the deeper
mathematical cause is that these histories are stored and reused as acceleration
cochains rather than as part of a conservative, common-flux update of `rho u`.

## Consequence

The next valid implementation target is not damping or pressure suppression.
It is a theorem-preserving update:

1. Extract the face mass flux from the same FCCD phase transport that updates
   `psi`.
2. Use that exact mass flux to update conservative momentum `m = rho u`.
3. Recover velocity from the transported mass and momentum.
4. Treat reinitialization/retraction as an operation on `(psi, rho, rho u)`, not
   only on `psi`.
5. Recast affine pressure/capillary history in a work/energy-compatible
   momentum form, or fail-close when the stored face-acceleration history
   violates a certified energy budget.

Only after this common-flux/momentum theorem is restored should capillary-face
conditioning and PPE residual tolerances be evaluated as secondary issues.
