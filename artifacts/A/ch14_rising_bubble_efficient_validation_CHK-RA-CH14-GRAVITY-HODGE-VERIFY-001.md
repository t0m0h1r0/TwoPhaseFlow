# CHK-RA-CH14-GRAVITY-HODGE-VERIFY-001

## Scope

Efficiently validate the variational-gravity Hodge implementation for the
water-air rising-bubble problem without spending the first cycle on a long
zero-start run.

The diagnostic question is whether the remaining blow-up band is caused by the
new variational gravity force, by surface tension, by checkpoint/restart
state, or by an inconsistent carried velocity/face state.

## Verification Strategy

The efficient test hierarchy was:

1. Production restart safety gate:
   attempt to resume an old hot checkpoint with the new YAML contract.
2. Diagnostic restart bypass:
   load the same hot arrays into the new solver without accepting it as a
   production restart, then selectively remove or zero state components.
3. Force isolation:
   repeat the hot continuation with normal `sigma,g`, `sigma=0`, `g=0`, and
   `sigma=g=0`.
4. State isolation:
   keep the same hot `psi/rho` but zero the velocity, conservative momentum,
   and projected-face state.
5. One-step map comparison:
   compare the current discrete map from the hot checkpoint with old projected
   faces preserved, projected faces cleared, and zero-velocity control.

This avoids interpreting a long run after an invalid scheme-changing restart
as physical evidence.

## Results

The production restart correctly failed closed:

```text
CheckpointError: checkpoint config fingerprint differs; only final time and
visualization/output paths may change
```

The diagnostic bypass showed that a hot state generated by the previous route
is not a valid continuation state for the new route.  With old histories reset,
the same hot velocity/momentum became unstable even when both surface tension
and gravity were removed:

```text
normal sigma,g       unstable near t = 0.02129
sigma=0, g=9.81     unstable near t = 0.02121
sigma=0.072, g=0    unstable near t = 0.02102
sigma=0, g=0        unstable near t = 0.02110
```

By contrast, keeping the same hot geometry but zeroing velocity, conservative
momentum, and projected faces passed to `t=0.024`; with normal force enabled it
also passed to `t=0.024`:

```text
normal force, zero velocity:
  t_final        0.024
  local_steps    566
  speed_linf     1.3194356659768314e-01
  div_u_max      7.4317423326419885e-09
```

The old checkpoint is internally consistent between nodal velocity and
conservative momentum, but not between nodal velocity and stored projected face
state:

```text
state_speed_linf                 8.101912244e-01
momentum_speed_linf              8.101912244e-01
state_vs_momentum_u_linf         2.78e-17

state_vs_face_reconstruct_v_linf 7.935084867e-02

```

The one-step map comparison from the old hot checkpoint gave:

## Interpretation

The old hot checkpoint must not be used as a production validation of the new
variational-gravity route.  The fail-close manifest check is therefore doing
the physically correct thing: it prevents an old discrete Hamiltonian/Hodge map
from being interpreted as a valid state for a different map.

The direct cause of the diagnostic blow-up is not isolated to the newly added
gravity covector.  The instability persists with `sigma=g=0`, while the same
geometry with zero velocity and zero momentum survives.  Therefore the strong
candidate is a carried high-energy velocity/momentum state that is compatible
with the old discrete map but not with the current common-flux, transported
mass, pressure-adjoint map.

The stored projected-face mismatch is real and should remain a restart
invariant to test, but it is not by itself the whole cause: clearing the old
face state while retaining the hot velocity increases the one-step pressure
RHS.  The physically meaningful diagnostic is therefore the combined state
compatibility of `(psi, rho, m, projected_faces, pressure/history)` under the
same discrete variational map, not the geometry alone.

## Next Verification Target

The efficient next production validation should be a zero-start run on the new
route with periodic rollback checkpoints, plus a same-map restart equivalence
check at an early time before the expensive blow-up band.  Reusing pre-change
hot checkpoints should remain diagnostic-only.
