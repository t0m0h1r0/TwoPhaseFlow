# v5.1 Concurrency-Aware — 2-Session Dry Run Checklist

Human-executed verification procedure for meta v5.1.0 (CHK-114) before flipping `concurrency_profile` from `"legacy"` to `"worktree"` in production (`prompts/agents/_base.yaml`). This file is documentation only — no automation, no Claude Code invocations from inside this document. A human operator runs the steps in two terminals and records the outcome in `docs/02_ACTIVE_LEDGER.md §4 BRANCH_LOCK_REGISTRY`.

## Prerequisites

- Branch `meta/v5.1-concurrency-aware` merged (or at least checked out in both terminals' primary worktrees).
- `jq` available (`brew install jq` or equivalent) — the schema lint uses it.
- Python 3 available (used by `LOCK-ACQUIRE` / `LOCK-RELEASE` snippets in `meta-ops.md`).
- No active lock on the test branches `meta-v5.1-demo-a` / `meta-v5.1-demo-b` — inspect `docs/02_ACTIVE_LEDGER.md §4` and `docs/locks/` before starting.
- **Flip the flag BEFORE starting**: `concurrency_profile: "legacy"` → `"worktree"` in `prompts/agents/_base.yaml`, on a throwaway branch or local-only change. If the dry run passes, commit the flip as a standalone one-line commit (Phase D.3). If it fails, discard.

## Test scenario overview

Two sessions (A and B) each do a minimal write cycle on their own branch, in isolated worktrees, acquiring and releasing their own locks. Then a third check verifies that a collision correctly triggers STOP-10.

```
Session A (Terminal 1)                 Session B (Terminal 2)
──────────────────────                 ──────────────────────
 GIT-WORKTREE-ADD                      GIT-WORKTREE-ADD
 ↓                                      ↓
 LOCK-ACQUIRE meta-v5.1-demo-a         LOCK-ACQUIRE meta-v5.1-demo-b
 ↓                                      ↓
 edit docs/memo/demo_a.md              edit docs/memo/demo_b.md
 ↓                                      ↓
 HAND-01 → HAND-02 cycle               HAND-01 → HAND-02 cycle
 ↓                                      ↓
 GIT-ATOMIC-PUSH (if remote)           GIT-ATOMIC-PUSH (if remote)
 ↓                                      ↓
 LOCK-RELEASE                          LOCK-RELEASE

Then: Terminal 1 attempts LOCK-ACQUIRE meta-v5.1-demo-b while Session B
still holds it → MUST get STOP-10. Terminal 1 MUST NOT overwrite.
```

## Detailed procedure

### Step 0 — Static gates (both terminals, before any lock)

```bash
# Axiom preservation (MUST all pass before starting)
grep -c "^## φ[1-9]:"        prompts/meta/meta-core.md     # must == 7
grep -c "^## A[1-9][0-9]*:"  prompts/meta/meta-core.md     # must == 11
grep -c "^### φ4.1:"         prompts/meta/meta-core.md     # must == 1
grep -c "^### A8.1:"         prompts/meta/meta-core.md     # must == 1

# Schema lint
jq empty prompts/meta/schemas/hand_schema.json

# _base.yaml parse + meta_version
python3 -c "import yaml; d=yaml.safe_load(open('prompts/agents/_base.yaml')); print(d['meta_version'], d['concurrency_profile'])"
# expected after flip:  5.1.0 worktree
```

If ANY static gate fails → abort dry run; fix before proceeding. These are the invariants that make v5.1 load-bearing.

### Step 1 — Session A worktree + lock

Terminal 1:
```bash
export SESSION_ID=$(python3 -c "import uuid; print(uuid.uuid4())")
export BRANCH="meta-v5.1-demo-a"
export BRANCH_SLUG=$(echo "$BRANCH" | tr '/' '-')
export HOLDER_AGENT="PaperWriter"
export WORKTREE_PATH="../wt/${SESSION_ID}/${BRANCH_SLUG}"

# From the primary checkout:
git branch "$BRANCH" main                        # fresh branch
git worktree add "$WORKTREE_PATH" "$BRANCH"      # GIT-WORKTREE-ADD

cd "$WORKTREE_PATH"
export LOCK_FILE="docs/locks/${BRANCH_SLUG}.lock.json"

# LOCK-ACQUIRE (copied verbatim from meta-ops.md §LOCK-ACQUIRE):
python3 - <<'PY'
import os, json, datetime, sys
path = os.environ["LOCK_FILE"]
payload = {
  "branch":        os.environ["BRANCH"],
  "branch_slug":   os.environ["BRANCH_SLUG"],
  "session_id":    os.environ["SESSION_ID"],
  "worktree_path": os.environ["WORKTREE_PATH"],
  "holder_agent":  os.environ["HOLDER_AGENT"],
  "acquired_at":   datetime.datetime.utcnow().isoformat(timespec="seconds") + "Z",
  "expires_at":    (datetime.datetime.utcnow() + datetime.timedelta(hours=24)).isoformat(timespec="seconds") + "Z",
  "meta_version":  "5.1.0",
}
try:
    fd = os.open(path, os.O_WRONLY | os.O_CREAT | os.O_EXCL, 0o644)
except FileExistsError:
    sys.stderr.write("STOP-10 foreign lock: " + path + "\n"); sys.exit(10)
with os.fdopen(fd, "w") as f:
    json.dump(payload, f, indent=2)
print("ACQUIRED", path)
PY

# Append a row to docs/02_ACTIVE_LEDGER.md §4 BRANCH_LOCK_REGISTRY
# (manual markdown edit; the automated wrapper would inject this)
```

Expected: `ACQUIRED docs/locks/meta-v5.1-demo-a.lock.json` printed. File exists. §4 in the primary checkout shows one new row.

### Step 2 — Session B worktree + lock (PARALLEL to Step 1)

Terminal 2, while Terminal 1 still holds its lock:
```bash
export SESSION_ID=$(python3 -c "import uuid; print(uuid.uuid4())")  # DIFFERENT UUID
export BRANCH="meta-v5.1-demo-b"
export BRANCH_SLUG=$(echo "$BRANCH" | tr '/' '-')
export HOLDER_AGENT="CodeArchitect"
export WORKTREE_PATH="../wt/${SESSION_ID}/${BRANCH_SLUG}"

git branch "$BRANCH" main
git worktree add "$WORKTREE_PATH" "$BRANCH"
cd "$WORKTREE_PATH"
export LOCK_FILE="docs/locks/${BRANCH_SLUG}.lock.json"

# Same LOCK-ACQUIRE block; must SUCCEED because it's a different branch_slug
```

Expected: `ACQUIRED docs/locks/meta-v5.1-demo-b.lock.json`. Both terminals now hold independent locks. `docs/02_ACTIVE_LEDGER.md §4` shows TWO active rows.

### Step 3 — Writes + HAND-02 cycle

Each terminal: edit one tiny docs file (e.g., `docs/memo/demo_a.md` / `demo_b.md`), compute a sha256 over it as `verification_hash`, construct a schema-valid HAND-02 envelope (`prompts/meta/schemas/hand_schema.json`), commit, push if remote is configured.

```bash
python3 - <<'PY'
import hashlib, json, os
content = open("docs/memo/demo_a.md", "rb").read()
envelope = {
  "hand_type": "HAND-02",
  "session_id": os.environ["SESSION_ID"],
  "branch_lock_acquired": True,
  "verification_hash": hashlib.sha256(content).hexdigest(),
  "timestamp": "2026-04-11T10:00:00Z",
  "payload": {"status": "SUCCESS", "produced": ["docs/memo/demo_a.md"], "issues": [], "lock_released": False, "stop_code": None},
}
print(json.dumps(envelope, indent=2))
PY
```

(The payload above will be flagged by strict validators because `stop_code: None` is not conformant — that is intentional for SUCCESS; the real schema lets stop_code be omitted.)

### Step 4 — GIT-ATOMIC-PUSH (if a remote is configured)

```bash
git fetch origin
git rebase origin/main    # rebase BEFORE push
git push origin "$BRANCH"
```

- Expected: clean push.
- If rebase conflicts → STOP-11 (SOFT). Agent would halt. For the dry run, `git rebase --abort`, note the trigger in §4, continue to Step 5 anyway.

### Step 5 — LOCK-RELEASE

```bash
python3 - <<'PY'
import os, json, sys
path = os.environ["LOCK_FILE"]
with open(path) as f:
    data = json.load(f)
if data.get("session_id") != os.environ["SESSION_ID"]:
    sys.stderr.write("STOP-10 foreign lock force: " + path + "\n"); sys.exit(10)
os.remove(path)
print("RELEASED", path)
PY

# Update §4 row: add released_at timestamp (DO NOT delete the row).
```

Expected: file removed, `RELEASED` printed, §4 row updated with `released_at`.

### Step 6 — Negative case (REQUIRED — this is the load-bearing test)

Terminal 1, while Terminal 2 still holds its lock (i.e., do this BEFORE Terminal 2 runs Step 5):
```bash
# Terminal 1 tries to acquire Terminal 2's branch lock
export SESSION_ID=$(python3 -c "import uuid; print(uuid.uuid4())")  # NEW uuid — pretending to be a foreign session
export BRANCH="meta-v5.1-demo-b"
export BRANCH_SLUG="meta-v5.1-demo-b"
export LOCK_FILE="docs/locks/${BRANCH_SLUG}.lock.json"
export HOLDER_AGENT="EvilImpersonator"
export WORKTREE_PATH="/tmp/will-not-be-created"

python3 - <<'PY'
import os
try:
    fd = os.open(os.environ["LOCK_FILE"], os.O_WRONLY | os.O_CREAT | os.O_EXCL, 0o644)
    print("UNEXPECTED SUCCESS — STOP-10 guard is BROKEN"); exit(1)
except FileExistsError:
    print("STOP-10 foreign lock — correctly blocked")
PY
```

Expected output: `STOP-10 foreign lock — correctly blocked`. If instead you see `UNEXPECTED SUCCESS` → the mutual exclusion is broken; abort the rollout and fix before flipping.

### Step 7 — Integration check (ExperimentRunner dry run)

Pick a trivial experiment (e.g., `experiment/ch11/exp11_2_dccd_filter.py` — fast), invoke ExperimentRunner with `--dry-run` or the minimum-cost run, confirm:
- SC-1..SC-4 emit their pass/fail individually.
- The agent emits a HAND-02 envelope whose `verification_hash` matches sha256 of the result package.
- The envelope validates against `prompts/meta/schemas/hand_schema.json`.

(This step depends on the actual E-Node implementation — under v5.1 it is procedural guidance from the agent file, not yet enforced by a runtime validator.)

## Recording results

Record in `docs/02_ACTIVE_LEDGER.md §4 BRANCH_LOCK_REGISTRY` (even though the test branches are temporary):

1. Step 1 ACQUIRE row for `meta-v5.1-demo-a` with released_at.
2. Step 2 ACQUIRE row for `meta-v5.1-demo-b` with released_at.
3. Step 6 note: "STOP-10 triggered as designed — mutual exclusion verified".
4. Step 7 note: "ExperimentRunner HAND-02 schema-valid: YES/NO".

Then update the CHK-114 body in §1 with the verification outcome. If all steps pass → Phase D.3 flag flip is approved. If any fails → document the failure, revert `concurrency_profile` to `legacy`, and fix before retrying.
