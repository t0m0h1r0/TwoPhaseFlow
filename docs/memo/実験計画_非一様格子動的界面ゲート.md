# 実験計画：非一様格子・動的界面ゲート

日付: 2026-04-16  
対象: `alpha_grid > 1` を本線へ戻す前提条件の定義

## 1. 目的
静止界面テストだけでは不十分である。  
界面が移動・変形する条件で、非一様格子が破綻しないことを「採用ゲート」として先に満たす。

## 2. ゲート基準（暫定）
各ケースで以下をすべて満たしたとき `PASS`:

1. 数値破綻がない（例外なし、非有限値なし）
2. 最終質量誤差 `|M-M0|/|M0| <= 1e-2`
3. 速度ピークが有限かつ `max|u| <= 1e3`

備考: 閾値はまず「破綻回避」を優先した暫定値。安定化後に厳格化する。

## 3. テスト行列
実装: `experiment/ch12/exp12_20_nonuniform_dynamic_gate.py`

1. `static_uniform_control`  
`alpha=1.0`, wall, `sigma>0`（対照）
2. `static_nonuniform_control`  
`alpha=2.0`, wall, `sigma>0`
3. `translate_nonuniform`  
`alpha=2.0`, periodic, `sigma=0`, 初期一様流速あり（移流で界面移動）
4. `capillary_nonuniform`  
`alpha=2.0`, periodic, `sigma>0`（動的曲率駆動）

## 4. 実行手順
ワークツリーで以下を実行:

```bash
make push
make run EXP=experiment/ch12/exp12_20_nonuniform_dynamic_gate.py
make pull
```

出力:
- `experiment/ch12/results/20_nonuniform_dynamic_gate/data.npz`
- `experiment/ch12/results/20_nonuniform_dynamic_gate/nonuniform_dynamic_gate.pdf`

## 5. 初回ベースライン（この計画に追記）
この節に初回実測を追記し、今後の修正で比較する。

記録項目:
1. `pass/fail`
2. `fail_step`
3. `mass_final`
4. `u_peak`

### 5.1 2026-04-16 実測（exp12_20 初回）

| Case | pass | fail_step | mass_final | u_peak | 判定 |
|---|---:|---:|---:|---:|---|
| static_uniform_control | 1 | -1 | 3.308e-04 | 1.956e-01 | PASS |
| static_nonuniform_control | 1 | -1 | 2.456e-03 | 6.684e+01 | PASS |
| translate_nonuniform | 0 | -1 | 1.554e-04 | 2.282e+18 | **FAIL** (速度暴走) |
| capillary_nonuniform | 1 | -1 | 2.546e-03 | 2.790e+01 | PASS |

初回ゲート結論:
1. 静止系と壁面毛管系は暫定閾値を満たす。  
2. 動的平行移流（非一様・壁）で `u_peak` が上限を大幅超過し、ゲート未達。  
3. 非一様格子の本線採用判断は、`translate_nonuniform` の暴走抑制を確認するまで保留。

## 6. 次アクション（fail時）
1. `fail_step` が浅いケース（早期破綻）を優先修正
2. `translate_nonuniform` と `capillary_nonuniform` を分離して原因切り分け
3. `metric`・`reinit`・`PPE coupling` の寄与をA/Bで固定して再計測

## 7. 原因切り分け進捗（2026-04-16, exp12_21）
診断実験: `experiment/ch12/exp12_21_nonuniform_translate_rootcause.py`

比較ケース（translate, sigma=0）:
1. `uniform_control`  
2. `nu_default`（rebuild=毎step, reproject=ON）
3. `nu_no_reproject`（rebuild=毎step, reproject=OFF）
4. `nu_no_rebuild`（rebuild=OFF）
5. `nu_rebuild10`（rebuild=10stepごと）

実測要約:
- `nu_default`: `u_peak=2.282e+18`, `div_peak=5.034e+18`
- `nu_no_reproject`: `u_peak=3.999e+07`, `div_peak=1.344e+08`
- `nu_no_rebuild`: `u_peak=6.201e+00`, `div_peak=1.982e+02`
- `nu_rebuild10`: `u_peak=8.426e+00`, `div_peak=3.645e+01`

暫定結論:
1. 主要因は「非一様格子の毎step再構成」による発散的増幅。  
2. `reproject_velocity` は悪化要因（ONで約10桁以上悪化）が、単独主因ではない。  
3. `rebuild_freq` を下げるだけで大幅改善するため、短期対策は「再構成頻度制御 + 再投影の安定化」が有効。  

次の修正候補（優先順）:
1. `alpha>1` での `grid_rebuild_freq` デフォルトを `1 -> 10`（または適応化）  
2. `_reproject_velocity` のPPEを可変密度に合わせる/強制クリップを入れる前に残差監視を追加  
3. `rebuild` トリガを `step固定` から `界面移動量閾値` に変更（不要再構成を抑制）

### 7.1 短期パッチ検証（2026-04-16）
`TwoPhaseNSSolver` に安全側デフォルトを導入:
- `alpha_grid > 1` かつ `grid_rebuild_freq == 1` のとき、内部的に `rebuild_freq=10` を採用

再実行（exp12_21）結果:
- `nu_default`: `u_peak=2.282e+18 -> 8.426e+00`
- `div_peak=5.034e+18 -> 3.645e+01`

解釈:
1. 毎step再構成の抑制だけで暴走は実用域まで低減。  
2. 短期運用として有効（ただし根治ではない）。  
3. 次段は `reproject_velocity` 安定化と適応的 rebuild トリガに進む。

### 7.2 `reproject_velocity` 変密度化A/B（2026-04-16）
目的:
1. 再投影を `rho=1` から `rho(psi)` へ変更した場合の効果を確認。
2. `exp12_21` 上で legacy と同条件比較。

実装状態:
1. `TwoPhaseNSSolver(reproject_variable_density=False)` を既定値（本線安全）。
2. `True` のときのみ再投影で `D((1/rho)grad phi)=div` を使用。

実測（exp12_21 追加ケース）:
1. `nu_default` (`legacy`): `u_peak=8.426e+00`, `div_peak=3.645e+01`
2. `nu_default_varrho`: `u_peak=7.819e+27`, `div_peak=2.316e+28`
3. `nu_rebuild10_varrho`: `u_peak=7.819e+27`, `div_peak=2.316e+28`

結論:
1. 単純な変密度化（Phase A単独）は本系では悪化し、採用不可。
2. `rebuild_freq=10` でも改善しないため、毎step再構成問題とは独立の不整合が残る。
3. 本質対策は「変密度化 + 演算子随伴整合 + 界面条件整合」の同時実装が必要。

### 7.3 `reproject_mode` スケルトン導入（2026-04-16）
実装:
1. `TwoPhaseNSSolver` に `reproject_mode` を追加:
   `legacy | variable_density_only | consistent_iim | consistent_gfm`
2. `consistent_iim/gfm` は Phase-1 として legacy フォールバック（警告付き）。

`exp12_21` 追試（追加ケース）:
1. `nu_default_consistent_iim`: `u_peak=8.426e+00`, `div_peak=3.645e+01`
2. `nu_default_consistent_gfm`: `u_peak=8.426e+00`, `div_peak=3.645e+01`

解釈:
1. スケルトン導入で既存挙動は非回帰（legacy同値）。
2. 次段は `consistent_iim` を実体化し，RHS jump 補正を有効化して評価する。

### 7.4 `consistent_iim` 実体化 v1（2026-04-16）
実装:
1. `consistent_iim` で IIM 補正候補 `delta_q` を再投影PPEへ注入する 2 段解法を導入。
2. 受理ゲートを追加し，候補が `div` を悪化させる場合は基準投影へフォールバック。
3. `consistent_iim` の基準投影は安定性優先で `legacy`（`rho=1`）とした。

再計測（exp12_21, local fallback 実行）:
1. `nu_default` (`legacy`): `u_peak=8.426e+00`, `div_peak=3.645e+01`
2. `nu_default_consistent_iim`: `u_peak=6.490e+00`, `div_peak=2.859e+01`

解釈:
1. `consistent_iim` v1 は少なくとも本ケースで legacy 比改善を示した。
2. ただし受理ゲートで一部候補は reject されており，jump 設計の改善余地が残る。
3. 次段は gate 発火率・受理後の安定性を可視化し，`translate_nonuniform` ゲートへ横展開する。

### 7.5 `exp12_20` 動的ゲートへの横展開（2026-04-16）
実施:
1. `exp12_20_nonuniform_dynamic_gate.py` に
   - `translate_nonuniform_consistent_iim`
   - `capillary_nonuniform_consistent_iim`
   の2ケースを追加。
2. remote 実行（`make cycle EXP=experiment/ch12/exp12_20_nonuniform_dynamic_gate.py`）。

主要結果:
1. `translate_nonuniform` legacy: `u_peak=8.426e+00`, `mass_final=6.568e-03`
2. `translate_nonuniform_consistent_iim`: `u_peak=6.490e+00`, `mass_final=7.208e-03`
3. `capillary_nonuniform` legacy: `u_peak=1.984e+01`, `mass_final=1.610e-03`
4. `capillary_nonuniform_consistent_iim`: `u_peak=3.604e+00`, `mass_final=1.352e-03`
5. 追加2ケース含め全ケース `PASS`。

解釈:
1. `consistent_iim` v1 は動的ゲート条件でも速度ピーク抑制に有効（特に capillary case）。
2. mass は同程度レンジで維持され，現閾値内。
3. 次段は受理ゲートの作動統計（受理率/棄却率）を保存して，改善が「常時受理によるものか／安全フォールバック込みか」を分離評価する。

### 7.6 受理ゲート統計の導入と計測（2026-04-16）
実装:
1. `TwoPhaseNSSolver` に `reproject_stats` を追加。
2. 記録項目:
   - `iim_attempts`, `iim_accepts`, `iim_rejects`, `iim_fails`
   - `iim_reject_divergence`, `iim_reject_nonfinite`
   - `iim_crossings_total`, `iim_crossings_accept`, `iim_crossings_reject`
3. `exp12_20` / `exp12_21` で統計をログ出力・`data.npz` 保存。

実測:
1. `exp12_21 / nu_default_consistent_iim`:
   - attempts=18, accepts=10, rejects=8, fails=0
   - reject_divergence=8, reject_nonfinite=0
   - crossings_total=2368
2. `exp12_20 / translate_nonuniform_consistent_iim`:
   - attempts=18, accepts=10, rejects=8, fails=0
   - reject_divergence=8, reject_nonfinite=0
   - crossings_total=2368
3. `exp12_20 / capillary_nonuniform_consistent_iim`:
   - attempts=18, accepts=18, rejects=0, fails=0
   - reject_divergence=0, reject_nonfinite=0
   - crossings_total=2318

解釈:
1. translate系では補正候補の一部が棄却されるが、最終指標は legacy より改善。
2. capillary系では候補が全受理され、大幅な `u_peak` 低減を示す。
3. 棄却は全て「非有限」ではなく「div悪化」起因であり、数値破綻より受理条件未達が主因。
4. 次段は reject ステップで `delta_q` の強度を自動減衰（line search的）し、reject→acceptへ変換できるか検証する。

### 7.7 減衰探索（line-search）導入結果（2026-04-16）
実装:
1. `consistent_iim` で候補が棄却される場合、`delta_q` を
   `0.5, 0.25, 0.125` に減衰して再評価。
2. 受理された減衰ケースを `iim_backtrack_accepts` で集計。

再計測:
1. `exp12_21 / nu_default_consistent_iim`:
   - `u_peak=7.952e+00`, `div_peak=2.730e+01`
   - attempts=18, accepts=18, rejects=0
   - backtrack_accepts=5
2. `exp12_20 / translate_nonuniform_consistent_iim`:
   - `u_peak=7.952e+00`, `mass_final=7.267e-03`
   - attempts=18, accepts=18, rejects=0
   - backtrack_accepts=5
3. `exp12_20 / capillary_nonuniform_consistent_iim`:
   - `u_peak=3.604e+00`, `mass_final=1.352e-03`
   - attempts=18, accepts=18, rejects=0
   - backtrack_accepts=0

解釈:
1. reject を 0 にでき、`consistent_iim` の運用安定性が向上。
2. translate 系では一部ステップで減衰受理が必要（=補正強度過大が主因）。
3. capillary 系は原強度で全受理され、現補正式は相対的に適合。
